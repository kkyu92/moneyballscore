## 제안 방향 (3 tier)

### Tier 1 — 최소 IaC (spec-only, 즉시 실행 가능, 코드 변경 X)

`docs/alerts/` 디렉토리 신규 생성:
- `docs/alerts/README.md` — alert taxonomy (level 정의, routing 원칙)
- `docs/alerts/rules/*.md` — rule 별 spec (trigger / severity / channel / owner)

각 rule spec 예시 (Markdown frontmatter):
```yaml
---
rule_id: silent-drift-predict
sentry_query: level:warning tag:silent_drift
severity: warning
threshold: 1 event / 5min
channels: [telegram-alerts, sentry-slack]
owner: predict-pipeline
runbook: docs/runbooks/silent-drift.md
---
```

**가치**: 새 팀원 온보딩 + rule 변경 review path 확보 (git blame 가능). 코드 변경 없음. Sentry UI 는 여전히 source of truth.

**risk**: 0 (문서만 추가).

**후속 트리거**: Tier 2 fire 시 spec ↔ Sentry rule 검증 script 대상.

### Tier 2 — sync script (rule → Sentry API 자동 반영)

`scripts/sync-sentry-alerts.mjs` 신규:
- `docs/alerts/rules/*.md` frontmatter parse
- Sentry API (`GET /api/0/organizations/{org}/alert-rules/`) 로 현 rule 조회
- diff → `PUT /api/0/projects/{org}/{project}/rules/{id}/` 로 spec 반영 (dry-run 옵션 필수)
- CI cron 안 매일 drift 검증 (spec vs 실제 Sentry rule mismatch → Sentry warning 발화)

**의존**: `SENTRY_AUTH_TOKEN` (organization scope + `project:write` scope). 이미 있음 (`~/.claude/skills/develop-cycle` 안 Sentry MCP 세팅 참조).

**가치**: rule 정의 = source of truth. UI 안 hand-edit 시 CI drift 감지. rollback = git revert.

**risk**: 2 (Sentry API 잘못 호출 시 prod rule 삭제 위험 — dry-run 강제 + 승인 gate 필요).

**후속 트리거**: Tier 1 spec 5+ rule 박제 후 (rule 수 임계 도달 시 자동화 ROI 발생).

### Tier 3 — alert level 통일 (코드 side captureMessage 71건 tag 표준화)

- `packages/kbo-data/src/notify/sentry-tag.ts` 신규 — `captureAlert(level, tag, meta)` wrapper
- 71건 captureMessage/captureException call site 을 `captureAlert` 로 통합
- 각 alert 에 `alert_rule_id` tag 필수 → Sentry rule matcher 안 정확한 grouping

**가치**: alert deduplication + rule spec (Tier 1) 과 자동 매핑. dashboard filter 가능.

**risk**: 1 (기존 event grouping 변경 — Sentry alert 폭발 위험, 롤백 시 grouping 회복).

**후속 트리거**: Tier 2 rule 정의 안 tag matcher 명시 후.
